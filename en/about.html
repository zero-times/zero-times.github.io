About

I work across product decisions, application delivery and the backend needed to ship.

Flutter is my main implementation path. Node.js, the WeChat ecosystem and automation fill the gaps a working product needs. I narrow the problem before writing code.

How I work

  • Clarify product positioning and core capabilities first, then decide on implementation.
  • State my boundaries clearly — no overpromising scope I cannot cover.
  • Project types, collaboration modes and contact points stay direct and accessible.

Scope

The work I can cover

Flutter-led product work

Interfaces, feature delivery and release-oriented implementation for products that need a practical cross-platform path.

Node.js as the missing backend layer

APIs, small admin tools and the minimum backend structure needed to ship a working version.

End-to-end delivery for small products

Best suited to projects that need clarity, speed and a manageable scope rather than a large platform build.

Good-fit projects

Collaboration tends to work best when your project fits these patterns

Turning an idea into a shippable version fast

Instead of a big upfront plan, I prefer defining the smallest real version that can be built and validated.

Existing products that need tightening or gap-filling

If you already have pages, content or code but the direction is scattered, I help restructure and prioritize first.

Where technical and product judgment overlap

I don't just own an isolated module. I work better when we can decide together what to build, what to defer and when to ship.

Workflow

How I prefer to move things forward

Define the product and the goal first

Lock down the product positioning, core pages and must-ship capabilities before deciding on visuals and implementation order.

Build the key pages and core flows first

Prioritize the core pages and critical user flows, close the validation loop, then decide what extras are worth the investment.

Ship as the primary progress marker

Every phase points to a shippable version. Delivery itself is the best progress signal — not an endlessly refined plan.
